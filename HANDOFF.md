## 一、项目背景

**时间：2026-06-05**

用户（wwk）有一个旧的 Claude Code Skill，放在 `D:\D_Project\GitHub_Project\AI代办\SKILL.md` 里。这个 Skill 已经用了一段时间，功能是"AI 代办任务管理"——用户通过自然语言跟 AI 对话，AI 理解后把任务写到飞书多维表格里，再定时推送早晚报。

但旧的 SKILL.md 过于庞大（14KB），把所有东西塞在一个文件里：数据模型、飞书 API 细节、UX 规则、Cron 配置全部混在一起。用户觉得它"过于庞大了"，想系统性地重构，做成一个切实可用好用的功能。

## 二、初始需求碰撞

用户说了一个核心需求：**AI 代办就是需要用自然语言去描述。** 用户不想学任何命令、填任何表单，像跟人说话一样说"帮我记一下后天交大物实验报告"就行了。

### 最初的存储方案讨论

用户一开始的想法是本地存储，但很快意识到一个问题：**用户不只是需要用 AI 去看代办，他可能有时候自己也会打开代办去看。** 于是联想到了最常用的飞书，飞书的多维表格比较适合干这件事，而且能生成看板视图。

最终形成的第一版方案是：
- 用户通过自然语言描述任务（或从别处复制粘贴）
- AI 自动分类整理
- 存入飞书多维表格
- 定时推送（早晚报）
- 用户随时可以自己打开飞书看

### 使用场景

95% 的使用场景在**手机上**——通过微信或飞书，经由 CCconnect 连接到本地 Agent。剩下的 5% 是在电脑上直接操作。用户特别强调：交互以短消息为主，回复要简洁清晰适合手机阅读。

## 三、系统逻辑的逐步深化

### 3.1 长短任务区分

用户提出了一个非常关键的分类思路：**任务和任务之间是不一样的。**

他日常遇到的任务大致分两类：

- **类型 A：事务性任务** — 导员指派、比赛任务、项目需求。有明确的开始、进度、完成或过期状态。
- **类型 B：生活清单** — 洗衣服、出门带什么东西。简单，做了就是做了，没做就是没做。

但用户进一步深化了这个分类，提出了**短任务 vs 长任务**的概念：

**短任务（单点任务）：**
- 事务型：导员指派、比赛任务、项目子任务
- 清单型：洗衣、购物
- 独立型：不隶属于任何长任务

**长任务（宏观持续任务）：**
- 永续型：高数学习、英语学习——没有明确的"完成"概念，永远在执行
- 项目型：大创项目、科研项目——有明确的开始和结束

用户特别强调了一个逻辑：**长任务可以下发子任务，即长任务下发一些短任务，或者说这些短任务组成了长任务。** 比如写高数作业可以归类到"高数学习"这个长任务下，做机械建模可以归类到"大创项目"这个长任务下。

### 3.2 标签 vs 关联记录

原有的方案是用飞书 Base 的"标签"（select 字段）来做长任务标记，但用户觉得不令人满意——看不到长任务的类型、看不到整体状态、不能折叠展开看子任务、不能汇总统计。

我推荐了**双表 + 关联记录**的方案：拆成"短任务表"和"长任务表"两张表，通过飞书 Base 的关联记录双向链接。用户觉得可以试一试。

### 3.3 长任务结项逻辑的一个关键纠正

我最初设计的逻辑是"项目型长任务的所有子任务都完成时，自动结项"。用户直接否定了这个想法——他说得非常清楚：

> **"长期任务就算可以结项，但是并不是做完所有子任务就一定会结项的。** 因为有可能我这个当前这个项目的所有子任务都认完做完了，但只是这个项目这段时间没有东西需要我做，但不代表这个项目结项了。来任务之后还是需要去给他做的。"

所以最终的规则是：**所有长任务行为统一，不自动结项。只有用户明确说"结项"，才结项。** 也不需要分永续型/项目型了，所有长任务状态一致。

### 3.4 优先级的多维度打分

用户要求优先级不能是单一维度（只看截止日期），必须是多维度评判，而且要能动态调整。

最终的五维度模型：
1. **时间紧迫度** 40% — 今天/过期最高，7天以上最低
2. **用户语气** 20% — "紧急/加急"加分
3. **领域权重** 20% — 不同分类的基础权重可配置
4. **沉寂提权** 10% — 待办状态挂了N天没动，逐渐加分，防止被遗忘
5. **依赖紧密度** 10% — 有前置任务要先做

动态调整策略经过了一轮修正：**不自动降级**（防止临近任务因语气误判被意外忽略），但是有一个降级特例——检测到用户主动推迟截止日期时，允许按新日期重新计算并下调。

### 3.5 归档逻辑的演化

一开始设计的是"次日凌晨自动归档"。但用户指出一个实际问题：电脑凌晨大概率是关着的。

**最终改为 "随手归档"——每次操作任务时顺手检查一遍，把 completed_at < today 的记录移到归档表。** 晚报生成前额外检查一次，确保当日完成记录不丢失。

### 3.6 一个关键设计：手动打勾的补完成时间

用户可能会在飞书表格里手动打勾完成，而不是通过 AI 告诉它。这时候 AI 怎么知道完成时间？

最初的方案是"AI 发现变更 → 补完成时间 = 今天"。但另一个 AI 给了一个补刀建议：如果用户在周五完成了，但 AI 直到周一才被唤醒，它会错误地把完成时间写成周一。

**最终方案：AI 读取飞书系统字段"最后更新时间"，提取实际日期填入完成时间。如果读不到，使用该任务截止日期的次日作为归档判断基准。**

### 3.7 无 DDL 任务的兜底

用户经常会说"有空的时候看一下那篇文献"这样没有明确截止日期的任务。时间紧迫度占 40% 的权重，没有截止日期时这个维度就空置了。

**兜底规则：无截止日期时，时间紧迫度固定计 0 分，由其余维度重新归一后决定，默认映射为 P3 或 P4。**

## 四、架构选择的反复

### 4.1 纯 Skill → 混合架构 → 纯 AI

这个演进过程很有意思：

**第一阶段：纯 Skill**
用户现有的方案，全部逻辑写在 SKILL.md 里。问题：太大、维护难、一致性差。

**第二阶段：Python CLI + Skill 薄层**
我建议把确定性逻辑（优先级计算、标签衍生、飞书同步）写成 Python CLI，SKILL.md 只做 NLP 理解 + 润色。

**第三阶段：回到纯 AI**
用户用了一句话让我重新思考：

> CLI 所做的事情，感觉纯用 AI 去替代也不会消耗多少东西，反而还可以做得更好。

我想了想，确实——优先级打分、长任务衍生、数据汇总这些对 LLM 来说就是几句话的事，比写 if-else 灵活得多。**而且用户用的 DeepSeek API 很便宜，不需要省这点调用量。**

**最终架构去掉了 CLI，只保留了定时推送脚本**（因为定时任务没有 AI 会话环境，需要脚本独立触发）。

### 4.2 飞书依赖是否必要

用户坦诚：飞书"打开看看"的需求没有那么急需。但他又说了一句很实在的话：

> 人本能习惯上可能会去看一些类似于 to-do list 的东西，今天有哪些任务，列一张表，前面是方框，做完了可以打勾，然后划掉的那种。

而且他补充了一个关键洞察：**做完一件事第一时间不是通知 AI，而是自己去勾掉。** 这是很多任务管理系统失败的原因。

所以系统需要两个并行的入口：AI 对话 + 自己打开看/勾。

飞书最终保留，因为它天然解决了"在手机上看 + 打勾 + 多端同步 + 通知"这几个最麻烦的事。

### 4.3 推送方案的演变

最开始用 CCconnect 的定时任务来做推送，但用户说 CCconnect 的定时任务跑不了。

然后改为 Windows Task Scheduler + Python 脚本调 DeepSeek API 润色 + cc-connect send 推送。

后来用户问"为什么一定要通过 CCconnect 推送"，我意识到可以用 lark-cli 直接发飞书消息。于是改为 lark-cli 直接推送。

用户测试发现飞书消息只有标题没有内容，又遇到 markdown 渲染问题。最终通过飞书 post 格式 + `<tag>md</tag>` 解决。

最后用户希望同时推送到微信，于是保留了双渠道：飞书 bot（富文本）+ 微信 cc-connect（纯文本）。

### 5.2 字段设计

活跃任务表字段（最终版）：
- ID（文本，T-xxx 自增）、标题、描述
- 状态（待办/进行中/已完成/已取消）
- 优先级（P0🔥/P1🔴/P2🟡/P3🟢/P4⚪）——这是用户看了一张参考图后确定的 emoji 样式
- 所属分类（学业/生活/班务/项目/其他）
- 标签（多选：报告/PPT/代码/文献/会议/设计/实验报告/考试/作业/数据处理/视频/其他）——从旧 Base 的任务主表借鉴过来的

- 截止日期、开始时间、完成时间、创建时间、预估时长、完成建议、来源、priority_source

长任务表：
- 名称、状态（未开始/进行中/已结项）
- 子任务（关联记录，双向链接到活跃任务表）
- 子任务总数、已完成子任务数（Lookup 统计字段）

**关于旧 Base 的任务主表：** 它有 17 个字段，比新表多了标签（多选）、开始时间、创建时间、更新时间、子分类、长期标签。用户觉得"标签（多选）"很好用，于是加到了新表里。子分类用户没看懂是干什么的，没加。

### 5.3 关键教训：优先级选项名变更导致旧数据丢失

在创建过程中，我们首先用 "P0"、"P1" 等纯文本选项名创建了优先级字段，然后用户改了主意（看了参考图后）想加 emoji。更新选项名后，**已有的记录中优先级字段值为 null**，因为选项 ID 变了，旧值不匹配新选项。需要手动更新每条记录。

这个教训：**设计阶段就应该把选项名确定下来**，后期修改选项名会导致数据不一致。

### 5.4 lark-cli 的使用经验

- `+table-create --fields` 的 JSON 中，text 类型不需要加 `style` 字段，否则会报 "Invalid discriminator value"
- `+field-create` 中 lookup 类型需要加 `--i-have-read-guide` 参数
- `+record-list --format json` 返回的数据结构与 markdown 格式不同，需要解析 `data.fields` 和 `data.data` 数组
- `.cmd` 文件在 Python 的 subprocess.run 中直接调用会报"找不到文件"，需要用 node 直接执行 `run.js` 或使用 `shell=True`
- 更新 select 选项时，`+field-update` 需要加 `--yes` 确认

## 六、推送脚本开发过程

### 6.1 遇到的坑

**Python subprocess 调用 lark-cli .cmd 文件失败**
在 Windows git bash 中，Python 的 subprocess.run 不能直接执行 .cmd 文件。最终方案：用 node 直接执行 `lark_dir/node_modules/@larksuite/cli/scripts/run.js`。

**DeepSeek API 返回的 \\n 变成文字**
AI 返回的文本中有 `\n` 作为换行符的转义，需要 `message.replace("\\n", "\n")` 处理。

**命令行参数中的换行符被截断**
用 `--markdown` 参数传多行文本时，换行符导致参数被截断。最终方案：先把内容写到临时文件，用 bash 的 `$(cat tempfile)` 读取传递。

**飞书 markdown 渲染**
最初用 `--markdown` 发送，遇到"只有标题没有内容"的问题。改用 post 格式 + `<tag>md</tag>` 包裹内容，终于正确渲染。

**飞书 post + bold 不可行**
尝试用 `{"tag": "bold", "text": "xxx"}` 来实现加粗，但飞书 API 报错说 bold 不是有效 tag。最终所有内容都放 `<tag>md</tag>` 里，用 markdown 的 `**加粗**` 语法。

### 6.2 标题大小的选择

用户试了 `#`（一级）、`##`（二级）、`###`（三级），最后选定 `###`：
- `#` 太大了
- `##` 还是有点大
- `###` 刚好合适
- `**加粗**` 跟正文字号一样大，用来做标题也可以

最终方案：段落标题用 `###`，任务名和优先级用 `**加粗**`，正文纯文本。

### 6.3 WeChat 推送

用 cc-connect 的 `send` 命令。需要指定 `--data-dir` 参数（因为 cc-connect 的配置在 `D:\D_software\cc-connect\data` 而不是默认路径）。

WeChat session token 可能过期，cc-connect 内部会 "retrying with fresh context_token"。当 token 过期时，send 命令可能返回 `ret=-2`，但 cc-connect 会在后台重试刷新。

### 6.4 Task Scheduler 兼容性

Task Scheduler 的环境变量跟命令行不一样。必须使用完整路径：
- Python: `C:\SoftWare\python\python.exe`（而不是 python）
- bash: `D:\D_software\Git\usr\bin\bash.exe`（而不是 /usr/bin/bash）
- cc-connect: `C:\Users\wwk\AppData\Roaming\npm\cc-connect.cmd`

定时任务的创建命令：
```bash
schtasks /create /tn "AI代办-早报" /tr "C:\SoftWare\python\python.exe -X utf8 D:\...\morning_report.py" /sc daily /st 08:00 /f
schtasks /create /tn "AI代办-晚报" /tr "C:\SoftWare\python\python.exe -X utf8 D:\...\evening_report.py" /sc daily /st 20:00 /f
```

## 七、飞书 post 消息格式参考

这是用户从飞书开放平台智能体那里获取的完整参考。**下次开发时务必使用这些格式，不要猜。**

### 支持的标签

| Tag | 用途 | 写法 |
|-----|------|------|
| text | 普通文字 | `{"tag":"text","text":"xxx"}` |
| at | @某人 | `{"tag":"at","text":"@李明","user_id":"ou_xxx"}` |
| a | 链接 | `{"tag":"a","text":"点我","href":"https://xxx"}` |
| hr | 分隔线 | `{"tag":"hr"}` |
| img | 图片 | `{"tag":"img","image_key":"img_xxx","width":300,"height":200}` |

### 在 `<tag>md</tag>` 中支持的语法

| 语法 | 示例 |
|------|------|
| **加粗** | 直接写 `**文字**` |
| *斜体* | 直接写 `*文字*` |
| ~~删除线~~ | 直接写 `~~文字~~` |
| 无序列表 | `- 项目\n- 项目` |
| 有序列表 | `1. 第一步\n2. 第二步` |
| 标题 | `# ~ #####` |
| 引用 | `> 引用文字`（部分支持） |
| 行内代码 | `` `代码` ``（部分支持） |
| 换行 | `\n` |
| emoji | 直接写 🔥✅ |

### 完整消息结构模板

```json
{
  "msg_type": "post",
  "content": {
    "zh_cn": {
      "title": "消息标题",
      "content": [
        [{"tag": "text", "text": "普通文字"}],
        [{"tag": "hr"}],
        [{"tag": "md", "text": "### 标题\n- 列表\n- 列表"}],
        [{"tag": "a", "text": "点我", "href": "https://xxx"}],
        [{"tag": "md", "text": "**加粗** 和 *斜体*"}]
      ]
    }
  }
}
```

**注意：** content 是一个数组，每个元素是一行。行内可以多个 tag 并列（比如 `[{"tag":"text","text":"普通"},{"tag":"md","text":"**加粗**"}]`）。

## 九、App 方向的完整想法

### 9.1 为什么想做 App

用户用了一段时间的"微信/飞书 → CCconnect → AI"方案后，觉得**记录门槛还是不够低**。他想的是：

> "能不能获取手机的某个按键或者组合键，按下去之后弹出来一个小 UI 界面，开始录音或者弹出输入框，把任务复制粘贴进去。"

他观察到 OPPO 系统的"小布记忆"有一个类似的功能——按下一个按键后开始录音，记录当时的想法。但小布记忆没有把代办这个点做好。

### 9.2 App 的架构思路

用户的核心想法：**手机只做 Agent 的壳子，不跑大模型。** 大模型能力通过调用 DeepSeek API 获得。这个壳子可以高度定制化，也可以做一个通用壳子，把 SKILL.md 的规则打包进去。

### 9.3 存储方案的定位修正

我之前过于强调飞书同步。用户纠正：
- **大部分用户只用手机做代办记录**
- **不一定非要跟飞书绑定**
- 用飞书唯一的原因是飞书的多维表格对数据呈现好
- 用户想在自己 App 里也做出同样的呈现效果
- 飞书同步作为一个可选功能保留给需要的用户

### 9.4 API 成本

### 9.6 UI 动画的追求

用户特别在意**丝滑的 UI 动画**。他描述了几种想要的效果：
- 录音框弹出来时 "Q 弹或者干脆利落"
- 修改代办时有很多动画反馈
- 安卓常常给人"不丝滑"的感觉，但他希望这个 App 是丝滑的

**技术判断：** Flutter 或 Jetpack Compose 做弹簧动画（spring animation）很容易，一个 `dampingRatio = 0.6` 就能实现 Q 弹效果。任务卡片滑出删除、勾掉后的划线动画、列表拖拽排序，这些都有现成的库可以用。

4. 悬浮球

### 9.8 播报方案

保留飞书 bot 作为备份渠道。

### 9.9 App 的三件核心事

用户自己总结的：

1. **输入** — 语音或者文本，自然语言任务录入
2. **AI 处理** — 深层次地把所有事情做好分类、归类、时间周期管理、生命周期管理
3. **输出** — 定时任务或用户查看，以更高效、更便捷、更有美感的方式呈现

### 9.10 确认机制的灵活性

用户提出：AI 处理后要不要给用户确认或修改的机会？最终决定做成**可开关的**。

- 有的用户不想再思考标签贴的对不对——关掉确认
- 有的用户收完东西就丢后台——关掉确认
- 有的用户处理完需要弹 UI 供快速选择——打开确认

放在设置里，用户自己选。

### 9.11 主页面的设计共识

用户打开 App 第一眼应该看到的是**今天要做的事**。

但是有一个矛盾：**怎么知道这个任务是今天需要做的？不能以截止日期来认定"今天要做"。**

解决方案：AI 推荐。不只是看截止日期，而是看"今天不做会有什么后果"——剩余天数 ≤ 预估所需天数 × 1.5 → 建议今天动工。

主页面只有一个清单，不分段不折叠。今天截止的标 `🔥`，过期的标 `⚠️`，其他干干净净一行一条。
